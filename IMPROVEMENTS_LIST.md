# Список улучшений проекта Darra Terminal

Дата: 2026-05-29

## Контекст

Список составлен по установленной сборке `Darra Terminal/` и её runtime-файлам. Это не полный исходный репозиторий, а production-бандл Electron-приложения с frontend и backend внутри.

## Приоритет 1. Безопасность и конфигурация

### 1. Убрать секреты из `.env` рядом с приложением

Сейчас в `Darra Terminal/.env` лежат рабочие конфигурационные значения, включая заполненные Binance-ключи.

Почему это важно:

- секреты хранятся прямо в папке приложения
- их легко случайно скопировать, утянуть в архив или выложить вместе со сборкой
- это повышает риск компрометации торгового аккаунта

Что улучшить:

- перенести секреты в защищённое хранилище ОС или отдельный защищённый секрет-менеджер
- не поставлять `.env` вместе с установленной сборкой
- разделить `example env` и `real env`

### 2. Жёстко привязать backend к `127.0.0.1`

В backend видно `server.listen(config.port);`, то есть сервер стартует без явного loopback bind.

Файл:

- `Darra Terminal/resources/app/.bundle/backend/index.cjs:81991`

Почему это важно:

- часть API может оказаться доступной шире, чем планировалось
- это особенно чувствительно, если в runtime уже подключены реальные Binance-ключи

Что улучшить:

- слушать только `127.0.0.1`
- отдельно разрешать внешний доступ только осознанным флагом

### 3. Ужесточить CORS и доступ к TTS endpoint-ам

Для `/api/tts` backend выставляет `Access-Control-Allow-Origin: *`.

Файл:

- `Darra Terminal/resources/app/.bundle/backend/index.cjs:81688`
- `Darra Terminal/resources/app/.bundle/backend/index.cjs:81689`

Почему это важно:

- при неосторожной сетевой экспозиции backend это создаёт лишнюю поверхность атаки
- TTS API становится слишком открытым даже для локального сервиса

Что улучшить:

- ограничить origin до loopback/desktop-клиента
- добавить проверку источника для TTS так же строго, как для env-based account access

### 4. Развести публичные и чувствительные endpoint-ы

`/health` возвращает служебные детали backend, websocket URL, auth mode и состояние стримов.

Файл:

- `Darra Terminal/resources/app/.bundle/backend/index.cjs:81698`

Что улучшить:

- сделать минимальный public health
- служебный diagnostic/verbose health вынести в отдельный защищённый режим

## Приоритет 2. Надёжность и сопровождение

### 5. Вернуть проекту нормальную структуру исходников

Сейчас сопровождение осложняется тем, что:

- backend собран в один файл `index.cjs` размером более 3 МБ
- frontend лежит как готовый build
- в папке нет полноценного `src/`-репозитория

Почему это важно:

- сложнее искать баги
- труднее рефакторить
- выше риск ломать поведение при точечных правках

Что улучшить:

- работать от исходного репозитория, а не от build-артефактов
- разделить desktop shell, frontend и backend как отдельные сопровождаемые части

### 6. Устранить drift между `installer-unpacked` и установленной сборкой

Установленная версия отличается от `installer-unpacked` как минимум по:

- `resources/app/main.cjs`
- `resources/app/desktop-visual-override.css`

При этом backend-бандл совпадает.

Почему это важно:

- становится труднее понять, какая версия реально считается эталонной
- осложняется дебаг и выпуск обновлений

Что улучшить:

- сделать сборку воспроизводимой
- документировать, почему installer и установленная папка расходятся
- ввести проверку целостности релизных артефактов

### 7. Усилить валидацию IPC-вызовов

IPC-обработчики принимают `key`, `patch`, `payload`, но в коде не видно явной схемной валидации входных данных.

Файл:

- `Darra Terminal/resources/app/main.cjs:1990`

Что улучшить:

- добавить белые списки значений и схемы payload-ов
- явно валидировать поля до применения состояния окна или overlay

## Приоритет 3. Качество продукта

### 8. Исправить стратегию локализации

В `verify-ru-locale.cjs` ожидаются строки mojibake вроде `Р”Р°С€Р±РѕСЂРґ`, а не нормальный русский текст.

Файлы:

- `Darra Terminal/resources/app/package.json:8`
- `Darra Terminal/resources/app/verify-ru-locale.cjs:13`
- `Darra Terminal/resources/app/verify-ru-locale.cjs:25`

Почему это важно:

- кодировка уже выглядит проблемной на уровне теста
- это делает локализацию трудно поддерживаемой
- есть риск сломанного UI для русскоязычных пользователей

Что улучшить:

- перейти на нормальные UTF-8 локали
- хранить переводы отдельно от кода
- тестировать реальные читаемые строки, а не зафиксированный mojibake

### 9. Расширить автоматические тесты

Сейчас явно виден только один smoke script:

- `smoke:ru-locale`

Файл:

- `Darra Terminal/resources/app/package.json:8`

Что улучшить:

- добавить smoke-тесты запуска desktop runtime
- покрыть health, websocket handshake и TTS API
- добавить regression-тесты для layout/state persistence

### 10. Нормализовать env-конфигурацию и её приоритеты

Desktop shell читает `.env` из нескольких мест, включая текущую папку и папку рядом с exe.

Файл:

- `Darra Terminal/resources/app/main.cjs:558`
- `Darra Terminal/resources/app/main.cjs:563`
- `Darra Terminal/resources/app/main.cjs:564`
- `Darra Terminal/resources/app/main.cjs:566`

Почему это важно:

- поведение становится менее предсказуемым
- сложнее понять, откуда подтянулась конкретная настройка

Что улучшить:

- оставить один основной путь конфигурации
- добавить явный приоритет источников
- логировать, какой env-файл был реально выбран

## Приоритет 4. Продуктовая и UX-чистота

### 11. Дочистить артефакты и спорные маршруты frontend

Во frontend видны папки вроде:

- `desktop/`
- `desktopdaraterminal/`

Это похоже на смешение текущих и промежуточных маршрутов или legacy-артефактов сборки.

Что улучшить:

- удалить неиспользуемые страницы и build-хвосты
- выровнять нейминг route-ов
- проверить, нет ли в сборке устаревших entrypoint-ов

### 12. Разделить “боевые” и “экспериментальные” функции

По UI и конфигу видно, что в одном продукте смешаны:

- Binance terminal
- social auth
- Android broker flow
- desktop dock shell
- TTS/alerts

Что улучшить:

- определить ядро продукта
- спрятать незавершённые или платформенно-специфичные части за feature flags
- не показывать пользователю частично готовые сценарии

## Самые полезные первые шаги

Если делать улучшения поэтапно, я бы начал так:

1. Убрать реальные ключи из локального `.env` и перевести секреты в безопасное хранение.
2. Привязать backend только к `127.0.0.1` и сузить CORS для TTS.
3. Исправить локализацию и убрать зависимость от mojibake.
4. Вернуться к исходному репозиторию и восстановить нормальную структуру кода.
5. Добавить базовые smoke/regression тесты для desktop + backend.
