## Project Summary

Darra / Scalp Station is a local-first Binance USDT-M futures decision system. It combines a backend market/risk engine, a Next.js realtime dashboard, and an Electron desktop shell for multi-window trading workflows.

Darra is not a terminal with a journal. Darra is a decision system with execution attached.

- Signal: backend-ranked Binance futures symbols with score, momentum, flow, volume, spread, liquidation, funding and risk context.
- Decision: Decision Inbox, unified signals, why-now context and explicit `ENTER`, `WAIT` or `SKIP` decision context.
- Execution: backend `order_intent`, `request_order_preflight`, paper mode, gated live/testnet path, order validation, order audit and paper position lifecycle.
- Review: Decision Review, Decision Replay, auto-review from Binance position observation, review statistics and supporting research.
- Knowledge: deterministic Trader Knowledge home with known links, unknown gaps, chain health, coverage, missing links, replay coverage, review completeness and evidence readiness.
- Supporting tools: Account, Risk, Research, Telemetry, desktop windows and backend lifecycle signals.

Architecture in one sentence:

```text
Binance streams / REST -> backend engines -> WebSocket snapshot/frame_patch/order messages -> frontend store -> React panels and Electron windows
```

Important rule:

- Heavy market, risk, order and financial calculations belong on the backend.
- React components should display backend-calculated state, collect user input and send typed WebSocket requests.

## Decision System Doctrine

Darra is not a trading terminal with analytics attached.

Darra is a Decision Operating System for Traders. Market events become structured decisions, structured decisions become trade reviews, trade reviews become reusable knowledge, and knowledge compounds into trading skill. AI is only the interface on top of that knowledge.

Traditional platforms:

```text
Data
-> Charts
-> Orders
```

Darra:

```text
Data
-> Signal
-> Decision
-> Execution
-> Review
-> Knowledge
```

Every new feature must strengthen one of these layers. If a feature does not improve Signal, Decision, Execution, Review or Knowledge, its priority is low.

### Core Chain

The most important chain in the system:

```text
Market Event
-> UnifiedSignal
-> TradeDecisionContext
-> OrderIntent
-> PositionLifecycle
-> DecisionReviewObject
-> Trader Knowledge Graph
-> Knowledge Assistant
```

Meaning:

- Market Event: market event detected from exchange, account, risk or persistence data.
- UnifiedSignal: normalized opportunity or warning generated by the system.
- TradeDecisionContext: the trader decision moment, including visible signal, risk and reason for `ENTER`, `WAIT` or `SKIP`.
- OrderIntent: explicit trading action requested by the trader.
- PositionLifecycle: full lifecycle of the position after execution.
- DecisionReviewObject: structured review of a completed trade or decision.
- Trader Knowledge Graph: accumulated trader behavior, mistakes, playbooks and outcomes.
- Knowledge Assistant: AI interface over persisted knowledge, not the source of truth.

### Primary Objects

| Object | Answers |
| --- | --- |
| UnifiedSignal | What happened? Why is it important? Why now? |
| TradeDecisionContext | What signal was available? What risk was visible? Why `ENTER`, `WAIT` or `SKIP`? |
| OrderIntent | What did the trader want to do? What size? What validation was applied? |
| PositionLifecycle | How was the position managed? What events and changes occurred? |
| DecisionReviewObject | What happened? What was correct? What was incorrect? What should improve? |
| Trader Knowledge Graph | What mistakes repeat? What playbooks work? What regimes fit this trader? |

### Architecture Rules

Implementation Rule:

```text
Any feature that touches execution must preserve or create:

UnifiedSignal
-> TradeDecisionContext
-> OrderIntent
-> PositionLifecycle
-> DecisionReviewObject

If the chain cannot be preserved, the feature is not ready.
```

Observability Rule:

```text
Every object in the chain must be reconstructable after restart.
```

The system must be able to answer:

- What happened?
- What was visible?
- What was decided?
- What was executed?
- What happened next?
- What was learned?

Knowledge Rule:

```text
Knowledge objects are first-class domain objects.
```

Signals, decisions, reviews, mistakes, outcomes and playbooks are persistent domain objects, not UI artifacts. UI may change. Models may change. Knowledge must survive both.

```text
Persistence is product architecture, not infrastructure.
```

## Beta Release Definition

Beta means the workstation is safe and coherent enough for controlled local/paper/testnet use, with live trading still gated and explicitly documented. Beta does not mean the system is institution-grade, fully polished, or ready for broad unattended live trading.

- P0.3-A1 `maxLeverage` source is implemented with backend check coverage, but requires configured Binance/testnet runtime smoke before beta release sign-off.
- P0.3-A2 `maxDailyLoss` backend implementation/check coverage is DONE with an authoritative current-day realized PnL ledger/query; runtime confidence remains PARTIAL until configured runtime smoke is completed.
- P0.8 can become DONE only after P0.3 gaps are closed and real configured testnet runtime smoke confirms REST, market WS, private WS and order preflight alignment.
- P0.4 Testnet / Live Environment Policy hardening is DONE for readiness policy; release/runtime remains PARTIAL until runtime smoke confirms the configured testnet stack.
- Order validation must remain backend-owned.
- Live trading gates must not be weakened.
- Safe-To-Add lifecycle is implemented as a real submit-control lifecycle and must remain backend-owned.
- Packaged desktop smoke must pass.
- Backend build must pass.
- Frontend build must pass.
- README must reflect the real code, not desired future behavior.

### Not Required For Beta

- custom workspace presets;
- full decomposition of `scalp-station-app.tsx`;
- advanced analytics perfection;
- perfect replay UX;
- mobile Android polish;
- visual redesign;
- full institutional-grade portfolio risk engine.

### Explicit Non-Goals For Beta

Beta should not try to:

- add more modules just to look more complete;
- make the live workspace more complex;
- move calculations into React;
- make live trading easier to enable;
- remove safety gates for convenience;
- combine everything into one giant rewrite.

## AI Modification Rules

- Identify touched files before editing.

- Run relevant build/check commands after code changes.

- Never read `.env*`.
- Never print secrets.
- Never weaken live gates.
- Never bypass backend order validation.
- Never move financial calculations into React.
- Never make frontend the source of truth for risk.
- Never silently change database schema without migration.
- Never edit generated artifacts as source.
- Never do broad rewrites without explicit scope.
- Never mark BLOCKED work as DONE without verified source.

### Contract Rules

- If `backend/src/types/messages.ts` changes, check `frontend/lib/types.ts`.
- If server messages change, check `frontend/store/use-screener-store.ts`.

- Keep fields optional/additive unless a breaking migration is explicitly requested.
- If execution behavior changes, preserve or create the chain `UnifiedSignal -> TradeDecisionContext -> OrderIntent -> PositionLifecycle -> DecisionReviewObject`.
- If the chain cannot be reconstructed after restart, the execution feature is incomplete.

### Safety Rules

- Live trading must remain disabled by default.
- Backend must reject unsafe live orders.
- Paper mode must remain the safe default.
- Missing source data for enabled risk limits must reject conservatively.
- UI may warn, but backend must enforce.

- Treat signals, decisions, reviews, mistakes, outcomes and playbooks as persistent domain objects, not UI-only artifacts.
- Do not add Knowledge Assistant or AI-style explanations unless they can be traced to structured evidence.
- Persistence is product architecture, not infrastructure.
